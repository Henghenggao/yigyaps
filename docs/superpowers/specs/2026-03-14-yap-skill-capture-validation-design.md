# Yap技能采集与验证系统设计规范

> **版本**: 1.0.0
> **日期**: 2026-03-14
> **状态**: 已确认，待实施
> **作者**: Co-Founders

---

## 概述

本文档定义YigYaps平台"Yap技能采集与验证"系统的核心设计。该系统是连接人类专家知识与AI Agent调用的关键管道，采用**迭代收敛循环**架构，参照Outlier.ai的采集体验，但在所有权模型和隐私保护上完全倒置：专家是IP持有人，不是数据供应商。

---

## 核心设计原则

1. **迭代收敛，不做线性门禁** — 采集与验证交替进行，每轮测试暴露知识缺口，缺口驱动新一轮提问，直到通过率超过阈值才可发布
2. **盲面试官架构** — LLM用于生成问题，永远看不到专家的答案，答案直接写入加密Vault
3. **诚实的信任分级** — 只承诺当前技术可交付的保护，路线图项目明确标注
4. **专家感受到"资产感"，而不是"打工感"** — 每一步UX都强调"这是我的Yap，我在塑造它"

## Section 1: 核心数据模型

### 1.1 技能包文件结构

```
SkillPackage/
├── skill.json          # 清单（名称/作者/类别/版本/统计）
├── knowledge/
│   └── corpus.jsonl    # 问答对语料库，每行一条QA记录（加密存储）
├── tests/
│   └── history.jsonl   # 验证历史（生成题 + AI答案 + 专家判定）
└── coverage.json       # 当前覆盖状态（各维度通过率，明文元数据）
```

### 1.2 QA记录结构（corpus原子单元）

```json
{
  "id": "qa_001",
  "question": "客户要求在3天内交付，但需求文档还不完整，你怎么处理？",
  "expert_answer": "先锁定最小可交付范围，明确哪些需求是必须确认的硬约束...",
  "tags": ["时间压力", "需求不确定", "客户管理"],
  "scenario_type": "边界决策",
  "complexity": "L2",
  "created_at": "2026-03-14T10:00:00Z",
  "source": "structured_interview"
}
```

**`source` 字段枚举值（规范定义，全局唯一）：**

| 值 | 含义 | 产生于 |
|----|------|-------|
| `structured_interview` | 结构化访谈 | 采集阶段第1-2轮 |
| `case_judgment` | 案例对比判断 | 采集阶段第3-4轮 |
| `scenario_simulation` | 场景模拟 | 采集阶段第5轮+ |
| `apprentice_correction` | 修正AI错误 | 验证阶段，专家批改⚠️/❌时 |

- `expert_answer` 在存储时使用AES-256-GCM加密（DEK + Shamir分片）
- `tags` 和 `scenario_type` 为明文，用于覆盖统计，不暴露内容
- tags由本地规则匹配（关键词+领域词表），不调用外部API，保证答案内容不外泄

### 1.3 验证记录结构（tests/history原子单元）

```json
{
  "id": "test_001",
  "source_qa_id": "qa_001",
  "variation_type": "condition_swap",
  "generated_question": "客户要求在3天内交付，但团队成员减半，你怎么处理？",
  "ai_answer": "基于Yap推理生成的回答...",
  "expert_verdict": "partial",
  "expert_correction": "还需要补充：先确认哪些成员是关键路径...",
  "created_at": "2026-03-14T11:00:00Z"
}
```

- `expert_verdict` 枚举：`correct` / `partial` / `wrong`
- `expert_correction` 非空时，系统创建**新的QA记录**（不修改原始记录）：
  - 新记录的 `question` = 本验证记录的 `generated_question`
  - 新记录的 `expert_answer` = `expert_correction` 内容（加密存储）
  - 新记录的 `source` = `"apprentice_correction"`
  - 新记录的 `tags` 和 `scenario_type` 从本验证记录的 `source_qa_id` 原始记录继承，再由本地NLP补充
  - 原始QA记录保持不变（保留知识溯源）

## Section 2: 采集流程（盲面试官架构）

### 2.1 每轮采集的数据流

```
Round N:

  [LLM端]
  输入：已覆盖场景标签列表 + 当前轮次主题
  不输入：任何专家答案内容
  输出：下一个问题（纯文本）

  [专家端]
  看到：问题
  输入：答案

  [系统处理]
  ① 答案 → AES-256-GCM加密 → corpus.jsonl（绕过LLM）
  ② 答案 → 本地NLP提取tags（不调外部API）
  ③ tags → 更新coverage.json场景统计
  ④ 更新后的统计（不含答案内容）→ 回传LLM → Round N+1
```

**关键不变量：** LLM的输入上下文中永远不包含 `expert_answer` 字段。

### 2.2 采集会话设计

- **每轮会话时长**：30分钟以内，产出5-10个QA对
- **可随时中断**：进度持久化，专家下次继续
- **提取方法标注**：系统记录每个QA对来自哪种提取方法
  - `structured_interview` — 结构化访谈（显式知识，覆盖高）
  - `case_judgment` — 案例对比判断（隐性知识，覆盖中）
  - `scenario_simulation` — 场景模拟（深层直觉，覆盖低但高价值）
  - `apprentice_correction` — 修正AI错误（验证阶段产生，持续补充）

### 2.3 LLM问题生成策略（按阶段递进）

```
第1-2轮（结构化访谈）：
  LLM收到：领域标签 + 复杂度目标
  生成：开放性框架问题（"你在X领域最核心的判断标准是什么？"）

第3-4轮（案例判断）：
  LLM收到：已覆盖场景类型 + 待填充的场景缺口
  生成：对比性问题（"情况A和情况B，你的处理方式有何不同？"）

第5轮+（场景模拟）：
  LLM收到：覆盖统计 + 未触达的边界条件
  生成：极端场景问题（"如果X条件同时消失，你的决策会怎么变？"）
```

## Section 3: 测试生成 + 验证循环

### 3.1 变形题生成（4种变换）

输入：原始QA对的 `question` 字段（不含答案）

| 变换类型 | 说明 | 示例 |
|---------|------|------|
| `condition_swap` | 替换关键条件 | "预算不足" → "时间不足" |
| `param_extreme` | 参数极端化 | "小团队" → "500人团队" |
| `negation` | 负向提问 | "你会怎么做" → "你绝对不会怎么做" |
| `scenario_merge` | 合并两个场景条件 | 题A条件 + 题B约束 → 新场景 |

变形题生成使用外部LLM，**但只传入题目（question字段），绝不传入答案**，安全可行。

### 3.2 验证界面（类Outlier体验）

```
┌──────────────────────────────────────────────────┐
│  场景：客户预算砍半，上线日期不变                  │
│                                                    │
│  Yap的回答：                                       │
│  "优先砍非核心功能，锁定MVP范围，与客户确认..."    │
│                                                    │
│  ─────────────────────────────────────────────   │
│   ✅ 准确      ⚠️ 部分准确      ❌ 错误           │
│  ─────────────────────────────────────────────   │
│                                                    │
│  [选择⚠️或❌后展开]                               │
│  请输入正确或补充答案：                            │
│  ┌──────────────────────────────────────────┐    │
│  │ 还应该提到：先做一个影响评估矩阵...       │    │
│  └──────────────────────────────────────────┘    │
│                                [提交并继续 →]     │
└──────────────────────────────────────────────────┘
```

- 专家修正内容自动进入corpus（`source: "apprentice_correction"`）
- 每次验证后通过率实时更新
- 通过率低于40%时，系统提示"建议先补充更多基础QA对再验证"

## Section 4: 收敛标准 + 发布门禁

### 4.1 两级发布门槛

**评分规则：5项指标均为硬性AND门禁，全部达标才能发布，不做加权平均。**

| 指标 | Beta发布 | Verified发布 |
|------|---------|-------------|
| QA对数量 | ≥15个 | ≥30个 |
| 场景类型覆盖 | ≥3种 | ≥6种 |
| 验证通过率 | ≥60% | ≥80% |
| 完整验证轮次 | ≥1轮 | ≥2轮 |
| 来源多样性 | 不要求 | ≥2种source枚举值 |

仪表盘展示的"72/100分"是**可视化进度指示**，计算方式：各项指标完成度的简单平均值（每项满分20分）。这个分数仅供专家判断还差多少，不作为发布的判定依据——判定依据是上表5项硬门禁。

### 4.2 技能质量仪表盘（专家视角）

```
我的Yap：[项目管理专家技能]  当前分数：72/100

场景覆盖：
  ✅ 时间压力决策     8个QA  通过率 85%
  ✅ 需求变更处理     6个QA  通过率 80%
  ⚠️ 跨团队协作       3个QA  通过率 60%  ← 建议加强
  ❌ 远程团队管理     0个QA  未覆盖      ← 盲区

[开始新一轮采集] [进入验证模式] [查看详细报告]
```

仪表盘数据来源：`coverage.json`（明文元数据），不解密任何答案内容。

## Section 5: 信任架构（可验证承诺）

### 5.1 现在交付的技术保护（可验证）

| 保护措施 | 实现方式 | 专家可验证手段 |
|---------|---------|--------------|
| 答案不进LLM | 盲面试官架构，代码开源可审计 | 技术文档 + 代码路径公开 |
| 平台无法单独解密 | Shamir(2,3)分片，专家持有Share2 | 专家可导出并持有密钥文件 |
| 访问可追溯 | 链式审计日志（`yy_invocation_logs`，**新建表，需migration**） | "我的保护"页面实时可查 |
| 反训练法律约束 | 平台使用强制前提条款 | 注册时必须点击确认（不是页脚） |

### 5.2 Shamir密钥分片分配

```
Share 1 → 平台数据库（用于日常推理解密）
Share 2 → 专家本地（浏览器安全存储 + 可导出文件备份）
Share 3 → 平台冷备份（物理隔离，仅在专家丢失Share2时走恢复流程）

重建DEK需要任意2/3 shares。
平台单独持有Share1+Share3，但Share3是冷备，日常不可访问。
专家撤销 = 删除Share2 → Yap永久不可解密。
```

**撤销保证的诚实边界：**

当前版本的撤销保护是**程序性承诺，不是纯技术保证**。平台在技术上持有Share1+Share3，理论上可以在收到撤销请求前先重建DEK。

当前的控制措施：撤销操作触发立即生效（不设冷却期），并写入审计日志。

未来加强路径（🔜）：Share3存入硬件安全模块（HSM），需专家签名才能访问，从程序性承诺升级为技术保证。在此之前，平台在反训练条款中明确承诺不绕过撤销，专家知情同意。

### 5.3 专家"我的保护"页面（必建UI）

```
┌──────────────────────────────────────────────────┐
│  我的Yap保护状态                                   │
│                                                    │
│  密钥状态：✅ 活跃   [导出密钥备份] [吊销此Yap]   │
│                                                    │
```

### 5.4 诚实路线图标注

```
现在交付（✅）：
  盲面试官 + 本地加密Vault + Shamir分片 + 审计日志 + 反训练条款

路线图中（🔜）：
  Embedding向量加密检索（6-12个月）
  零知识验证报告（待技术成熟）

  密文空间推理（同态加密，2-3年外）
  任何当前技术做不到的密码学保证
```

## 技术约束总结

- **外部LLM调用限制**：只用于（1）生成问题、（2）生成变形题题目——两者均不包含 `expert_answer`
- **本地处理**：tags提取、通过率计算、仪表盘渲染均在平台服务端完成，不调外部API
- **向后兼容**：现有 `rules/` 目录结构保留，新系统在 `knowledge/corpus.jsonl` 之上叠加，不破坏现有CLI工作流
- **渐进式门槛**：60分Beta可发布，防止专家因完美主义而永远不上线

## 实施范围

本设计文档定义核心内核，供CLI通道和Expert Studio通道共用：

- **CLI通道**：`yigyaps capture`（新命令）驱动采集，`yigyaps validate-skill`（新命令）运行验证循环。现有17个命令均保持不变，新命令叠加。
- **Expert Studio**：Web界面呈现仪表盘、验证UI、"我的保护"页面
- **共用内核**：数据模型、盲面试官流程、Shamir密钥管理、审计日志

**新建数据库表（需migration）：**
- `yy_invocation_logs` — 审计日志表（skill_id, caller_agent, event_type, timestamp, prev_hash链式校验）
- `yy_skill_corpus` — QA对持久化（corpus.jsonl的数据库镜像，便于查询覆盖统计）
- `yy_skill_tests` — 验证历史（history.jsonl的数据库镜像）
